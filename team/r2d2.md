## AgentOps / AI delivery bridge (meta-role)

- **Goal**: Reduce **toil** and speed delivery by letting agents handle repetitive analysis—**without** sacrificing **safety** or **auditability**.
- **Shift**: Prefer **durable policies** (who can merge, what can deploy, required checks, HITL gates) over **one-off** pipeline spaghetti for every edge case—while still automating the **high-value** paths.

### Runbook: Cloudflare Tunnel (Raspberry Pi — Internet + LAN)

Use this when the **API** is reachable on a public hostname (example: `tagly-backend.brandstaetter.rocks` → tunnel → host port **8008**) and the **frontend** is on another hostname or port (tunnel to **5173** or a static build). The SPA calls the API with **cookies** (`withCredentials`), so **CORS** and **CSRF trusted origins** must list every **browser origin** that loads the UI.

#### 1. Backend (Django) — `.env` / Compose `environment`

| Variable | Purpose |
|----------|---------|
| `ALLOWED_HOSTS` | Must include the **API** hostname: `tagly-backend.brandstaetter.rocks` (comma list). Keep `localhost` / LAN IP only if you still access Django directly for debugging. |
| `SERVER_URL` | Public **canonical** URL of the product (usually the **frontend**), e.g. `https://tagly.brandstaetter.rocks` — used for links/emails, not as the API base. |
| `CORS_ALLOWED_ORIGINS` | Every origin the browser may use when calling the API: **`https://<frontend-host>`** (Cloudflare), plus optional LAN/dev entries, e.g. `http://192.168.x.x:5173`, `http://localhost:5173`. Use **https** for public hostnames. |
| `CSRF_TRUSTED_ORIGINS` | **Same** origins as `CORS_ALLOWED_ORIGINS` (Django CSRF for session login). Include `https://tagly-backend.brandstaetter.rocks` if users open the API or admin on that host. |

Production hygiene: `DJANGO_DEBUG=0`, strong `DJANGO_SECRET_KEY`, no secrets in logs.

#### 2. Frontend — `VITE_API_URL`

Set to the **public API base URL** (must match what the browser can reach):

`VITE_API_URL=https://tagly-backend.brandstaetter.rocks/api/v1`

Rebuild the frontend after changing this (`npm run build`) or set the same in the Compose `frontend` service env for dev. **LAN-only** builds can keep `http://<pi-ip>:8008/api/v1`; **Internet** builds must use the **https** tunnel URL.

#### 3. Cloudflare Tunnel ingress

- One route: `tagly-backend.brandstaetter.rocks` → `http://127.0.0.1:8008` (or Docker-published port on the Pi).
- Second route: public **frontend** hostname → `http://127.0.0.1:5173` (Vite dev) or → static file server / reverse proxy if you serve `frontend/dist`.

**Vite dev (`npm run dev` / Compose `frontend` service)** rejects requests whose `Host` header is not allowlisted. Accessing `https://tagly.brandstaetter.rocks` via Cloudflare sets `Host: tagly.brandstaetter.rocks` — you must allow that hostname. This repo sets `server.allowedHosts` in `frontend/vite.config.ts` (includes `tagly.brandstaetter.rocks` and `.brandstaetter.rocks`). For extra hostnames without editing code, set **`VITE_ALLOWED_HOSTS`** (comma-separated) in the environment that starts Vite. If you still see “Blocked request / not allowed”, confirm you pulled the latest `vite.config.ts` and restart the frontend container/process.

#### 4. HTTPS and Django behind the tunnel

Clients use **HTTPS** at Cloudflare; the origin may see **HTTP**. If redirects, CSRF, or `request.is_secure()` behave wrongly, configure **forwarded headers** on the Django side (e.g. `SECURE_PROXY_SSL_HEADER`, `USE_X_FORWARDED_HOST`) — add explicitly in `settings.py` when needed; do not guess in production without verifying headers from `cloudflared`.

#### 5. Troubleshooting

- **CORS errors**: Missing or wrong `CORS_ALLOWED_ORIGINS` / scheme (`https` vs `http`).
- **403 CSRF**: Add the **exact** frontend origin to `CSRF_TRUSTED_ORIGINS`.
- **Login works on LAN but not via Internet**: Cross-hostname **session cookies** may need stricter cookie flags (`SameSite` / `Secure`) — treat as a follow-up in application settings with **Vader** if env-only fixes are not enough.

#### 6. Git hygiene on the Raspberry Pi (pull / deploy hosts)

**Goal**: Only **application source** comes from GitHub; **machine-local config, runtime state, and data** stay out of the repo (already covered by `.gitignore` where applicable).

| Topic | Policy |
|-------|--------|
| **Secrets & config** | Use **`.env`** on the Pi (already gitignored). Never commit `.env`. Document keys in `.env.example` only. |
| **Celery Beat** | The file **`celerybeat-schedule`** (under `backend/` when Beat runs from there) is a **runtime schedule database**, not source. It is **gitignored**; do not add or commit it. If `git pull` complains it would overwrite an untracked `celerybeat-schedule`, remove or move it aside, pull, then let Beat recreate it. |
| **`package-lock.json`** | Stays **tracked** in the repo (lockfile for reproducible installs). If the Pi shows “local changes would be overwritten” on pull, you usually want **upstream’s** lockfile: e.g. `git checkout -- frontend/package-lock.json`, then `git pull`. Avoid committing Pi-only drift unless you intentionally regenerate the lockfile on a dev machine and merge via PR. |
| **Postgres / uploads** | DB files live in **Docker volumes** or host paths **outside** the clone — not in Git. Same for `media/` (gitignored). |
